JavaScript Event Listeners

Jan 23, 2018
Intro

When JavaScript was first introduced to me it was explained as any movement or action that is taking place on the screen (yes I know you can do movements/animations with CSS but stick with me). Think about a simple button that shows an alert saying hello when you click it. This is being done with event Emitters and Listeners in JS! You will be using this everywhere in your code so today we are going to talk about what an Event Emitters and Listeners are, how to use them, we will understand a little more on event bubble up effect and then we will look at event.target

Definitions
Event Emitters

First let’s talk about an event real quick. An event is something that the browser does or a user does. For example, when a browser finishes loading, that is a browser event. When the user clicks a button or adds text in an input, that’s a user event. An event Emitter is an event that is triggered in the application. Try not to over complicate the idea of emitters. The english language definition of emit is : verb - produce and discharge (something, especially gas or radiation). An event is emitted (produced), then you can program a series of actions that will happen after the event is emitted. For example: when that user clicks a button, basically the JS engine is saying ‘Hey over here!! A button was clicked (emitted).. Look at me!’. Then it is up to us as the JS programmers to listen to that event and program what happens next.

Event Listeners

Event listeners will take place all over your code. Event Listeners literally just hang around listening for an event to happen. So from the button click above, the click event is emitted saying: “‘Hey over here!! A button was clicked.. Look at me!’” and the event listener is waiting around listening for that event then says: ‘Okay button I hear you, now I am going to do whatever the programmer told me to do. Thanks!’ The event listener can be programed to listen to any general event or a specific event based off the class, name or ID. In pseudo code it sounds like this: Hey document, get the element with the ID of “myButton” and add an event listener of “click” to it. Then when that specific button is clicked I want you to alert the user “Hello World”. Here is the syntax for that program: document.getElementByID(“myButton”).addEventListener(“click”, sayHello(){ alert(‘Hello World’)}).

You might see examples online with the event handlers in the HTML like: <button onclick="alert(‘Hello World’)">Click Me</button> . This is called an on-event handler, and it is probably coming from an older example/article. This is no longer considered best practice because each object can only have one of these types of events. Also it can become inefficient and unmanageable pretty quickly. We want to keep these event handlers and listeners in our separate JavaScript files.

Below are a couple common event listeners. Go ahead and try them out in your editor by adding a console.log() or an alert().

Bubbling

Event bubbling is simple if you have a basic understanding of HTML elements. When a generic event is triggered it bubbles up through all of its parents elements until it reaches the document object. Take a look at my CodePen, read through the code and click the button to see what happens. Then click the green div and then just click the outer black div. What you should have noticed is that when you click the button, the alert is triggered on the button, then the green div, then the black div. The event bubbles up through all three elements. If you only click the black div, you will only see the alert from the balck div since that is the top of the document object.

Target Event Property

The target event property can be a very powerful tool. It will return the element that triggered the event instead of the event listener that triggered the event. Let go back to the codepen to see the difference. Click here. In Example #1 we are clicking on a paragraph with the ID of paragraphOne, and then sending an alert with a message. This should be familiar from previous examples. Now go to Example #2 and notice that in my function I am passing an event and want my alert to tell me the nodeName on the target of the event that I clicked. So in this case I get my message plus : ‘P’ which is the nodeName.

As you get into more complex applications you will be working with dynamic content where you won’t always know the ID name. Or you might want to grab something by a class name. In Example # 3 I want to target the elements with the class name ‘purple’. Unlike IDs, elements can have multiple class names and multiple elements with the same class name. What my code is saying is it wants to listen to a click event on anything on the document that contains the class ‘purple’.
